# CloudFront로 사용자 가까이서 응답하기

목표: CDN이 왜 필요한지 물리적으로 이해하고, 엣지에서 캐시 히트/미스가 어떻게 갈리는지 그릴 수 있다

- CDN이 필요한 이유 — 물리적 거리 = 지연(latency)
    
    핵심은 빛도 광케이블 안에서는 유한한 속도라는 점, 거리가 멀수록 hop도 늘고 패킷 손실·재전송 확률도 올라 간다. 지연에는 거리(propagation delay)와 처리(서버 응답)가 섞여 있는데, 오리진 서버를 아무리 튜닝해도 거리에서 오는 지연은 못 줄이니 CDN을 사용한다. CDN은 "콘텐츠를 사용자 근처로 미리 옮겨두자"로 이 거리 문제를 푸는 것.
    
- 엣지 로케이션 vs 리전 vs 오리진 위치 관계
    
    오리진(Origin) — 원본이 있는 곳. S3 버킷이나 ALB, 혹은 우리 EC2. 진실의 원천(source of truth)이고 보통 한 리전에 위치
    
    리전(Region) — 오리진이 사는 AWS의 큰 데이터센터 묶음(예: ap-northeast-2 서울). AZ들의 집합
    
    엣지 로케이션(Edge Location) — CloudFront가 캐시를 들고 있는 전 세계 수백 개 지점. 리전보다 훨씬 많고 사용자에게 더 가깝다. 사용자는 항상 가장 가까운 엣지로 라우팅된다.(Route 53의 anycast/지연 기반과 연계).
    
- Cache Hit / Miss 흐름 — 엣지에 있으면 즉시, 없으면 오리진까지
    
    요청이 엣지에 도착하면 두 갈래로 나뉜다.
    
    - Cache Hit: 엣지에 유효한(만료 안 된) 사본이 있는 경우. 오리진까지 가지 않고 즉시 응답하므로 빠르고 오리진 부하가 없다.
    - Cache Miss: 사본이 없거나 만료된 경우. 엣지가 리저널 엣지 캐시를 거쳐 오리진에서 받아온 뒤 자기한테 저장하고 응답한다. 첫 요청은 느리지만 이후 같은 요청은 Hit이 된다.
    
    확인 방법은 응답 헤더 `X-Cache` 값(`Hit from cloudfront` / `Miss from cloudfront`).
    
- TTL과 캐시 만료 — 얼마나 오래 들고 있을지
    
    TTL은 엣지가 사본을 "신선하다"고 믿는 기간이다. 적용 우선순위는 다음과 같다.
    
    1. 오리진이 보내는 `Cache-Control: max-age=N`(또는 `Expires`)
    2. 없으면 CloudFront의 Default TTL
    3. Minimum/Maximum TTL이 상·하한을 강제 (예: 오리진이 `max-age=86400`을 보내도 Maximum TTL이 3600이면 1시간으로 잘림)
    
    TTL이 만료돼도 바로 버리지 않는다. 다음 요청 때 `If-None-Match` / `If-Modified-Since`로 조건부 재검증하고, 오리진이 `304 Not Modified`를 주면 본문 없이 TTL만 갱신해 대역폭을 아낀다.
    
    Route 53 TTL(DNS 레코드 캐시 기간)과는 계층이 다른 별개 개념. 이건 콘텐츠 캐시 기간이다.
    
- 캐시 키 — 어떤 기준(URL/헤더/쿼리스트링)으로 같은 요청으로 보는가
    
    엣지가 "동일한 요청"이라고 판단하는 기준이 캐시 키다.
    
    - 기본값: 도메인 + URL 경로
    - 선택 포함 가능: 쿼리스트링(`?id=1` vs `?id=2`), 특정 헤더(`Accept-Language`로 언어별 분리), 쿠키(세션별 분리)

핵심 질문: "정적 이미지는 캐시가 잘 되는데 로그인한 사용자별 페이지는 왜 캐시하면 안 되는가?"

캐시의 전제는 "하나의 사본을 여러 사용자가 공유"이다. 그래서 캐시 가능 여부는 이렇게 나뉜다

- **공유 가능(캐시 O)**: 모두에게 동일한 콘텐츠 — 이미지, CSS, JS, 공개 게시물
- **공유 불가(캐시 X)**: 사용자마다 다르거나 실시간이어야 하는 콘텐츠 — 마이페이지, 장바구니, 잔고, 주문내역

실무 표준 구조는 둘을 분리하는 것, 정적 자산(`/static/*`, 이미지)은 CloudFront가 캐시하고, 동적·개인화 응답(`/api/*`, `/mypage`)은 캐시 정책을 `CachingDisabled`로 두거나 `Cache-Control: private, no-store`로 오리진까지 그대로 흘려보냅니다. CloudFront에서는 경로 패턴별 Behavior로 이걸 나눕니다.
