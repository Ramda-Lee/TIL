# Internet Gateway와 퍼블릭/프라이빗 서브넷

목표: VPC의 정문이 어디 있는지, 퍼블릭과 프라이빗 서브넷이 무엇으로 갈리는지 정확히 정의한다

퍼블릭 서브넷 vs 프라이빗 서브넷의 본질적 차이

- 둘은 IP 대역도, 만들 때의 설정도 동일하지만 라우트 테이블 차이가 있다.

IGW(Internet Gateway) — VPC와 인터넷을 잇는 정문

- IGW는 VPC에 1개 attach되는 논리적 게이트웨이로 아래 두가지 일을 진행한다.
    - 인터넷 ↔ VPC 트래픽 통로
    - 1:1 NAT 수행: EC2는 자기 퍼블릭 IP를 모르고, OS의 NIC에는 프라이빗 IP만 있고, IGW가 나가는 패킷의 출발지 IP를 퍼블릭 IP로 변환한다.

"퍼블릭 서브넷 = 라우트 테이블에 IGW로 가는 경로가 있는 서브넷" 정의

- 퍼블릭 서브넷은 라우트 테이블에 0.0.0.0/0 → IGW 경로가 있는 서브넷 프라이빗 서브넷이다.

라우트 테이블과 IGW의 관계 — 무엇이 서브넷의 성격을 결정하는가

- 라우트 테이블에 항상 10.0.0.0/16 → local 같은 VPC 경로가 기본으로 있고, 여기에 IGW 경로를 추가하는 순간 그 서브넷은 퍼블릭이 된다.

퍼블릭 IP 자동 할당 설정과 IGW의 관계

- 인터넷이 되려면 두가지가 다 필요하다.
    
    
    | 조건 | 역할 |
    | --- | --- |
    | 퍼블릭 IP (자동 할당 or EIP) | 인터넷에서 나를 식별할 주소 |
    | 라우트 테이블의 IGW 경로 | 패킷이 실제로 나갈 길 |

핵심 질문: "EC2에 퍼블릭 IP를 붙였는데도 인터넷이 안 되는 이유는 무엇인가?" (라우트 테이블 누락)

- 퍼블릭 IP는 주소일 뿐 경로가 아니라서 그렇다. 해당 서브넷의 라우트 테이블에 0.0.0.0/0 → igw-xxx 같은 항목이 없다면, 패킷은 VPC 내부의 local 경로밖에 모르니 인터넷행 패킷은 갈 곳이 없어 드롭된다.
예를 들어 EC2 SSH가 안된다면 점검 순서가 라우트 테이블 IGW 경로 → 퍼블릭 IP 유무 → 보안그룹 체크 순서이다.
