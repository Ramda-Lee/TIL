# 연결 드레이닝과 무중단 배포

즉시 제거의 문제 (진행 중 요청 502/504로 끊김)

- 로드밸런서가 백엔드 서버를 풀에서 즉시 빼버리면 발생하는 일
    
    ```bash
    시점 T0:  클라이언트 → LB → Server A (요청 처리 중, 응답까지 3초 걸림)
    시점 T1:  운영자가 Server A를 풀에서 제거 (배포/스케일 인)
    시점 T2:  Server A 프로세스 종료
    시점 T3:  클라이언트는 끊긴 연결로 502/504 받음
    ```
    
    - 502 Bad Gateway를 받을 경우: LB가 백엔드로 요청을 보냈는데, 백엔드가 연결을 끊거나 비정상 응답을 줌. TCP RST 받았을 때 흔히 발생
    - 504 Gateway Timeout: LB가 백엔드 응답을 기다리가 타임아웃. 백엔드 프로세스가 죽었거나 응답을 못 보내는 상태

Graceful shutdown의 의미

- 신규 요청은 거부하지만 진행 중인 요청은 끝까지 처리의 의미
    
    ```bash
    T0:  종료 신호 수신
         ├─ 새 연결 수락 중단 (listen 소켓 close)
         ├─ 헬스체크에 "unhealthy" 응답 시작
         └─ 기존 연결의 in-flight 요청은 계속 처리
    
    T0+a: LB가 헬스체크 실패 감지 → 풀에서 제외
          (이 시점부터 신규 트래픽이 다른 서버로 라우팅됨)
    
    T0+b: 진행 중 요청 모두 완료
    
    T0+c: 프로세스 정상 종료
    ```
    

nginx reload의 graceful한 동작 원리 (기존 워커 유지)

```bash
[reload 전]
master process (PID 1000)
  ├─ worker process (PID 1001)  ← 요청 처리 중
  ├─ worker process (PID 1002)  ← 요청 처리 중
  └─ worker process (PID 1003)  ← 요청 처리 중

[nginx -s reload 실행 = master에 SIGHUP]

1. master가 새 설정 파일 파싱 및 검증
2. 새 worker 프로세스 fork (PID 2001, 2002, 2003)
   → 새 worker는 새 설정으로 신규 요청 받기 시작
3. 기존 worker에게 "graceful shutdown" 신호 전송
4. 기존 worker는:
   - 새 연결 받지 않음
   - 진행 중 요청만 마저 처리
   - 끝나면 자진 종료
5. 모든 기존 worker 종료 → reload 완료
```

Spring Boot graceful shutdown 설정 (server.shutdown=graceful)

```bash
# application.yml
server:
  shutdown: graceful

spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s  # 기본값

# 동작 방식
# SIGTERM 수신 시
1. DispatcherServlet이 새 요청 거부 시작
   (단, 임베디드 톰캣은 새 connection accept를 중단)
2. 이미 받은 요청은 계속 처리
3. timeout-per-shutdown-phase (기본 30초) 안에 끝나야 함
4. 시간 초과 시 강제 종료
```

AWS Deregistration Delay 개념 (기본 300초)

- ALB/NLB의 Target Group 설정에 있는 `Deregistration delay` (기본 300초)가 바로 connection draining

```bash
[배포 시작]
T0:    ECS가 새 Task 띄움 → ALB에 register, 헬스체크 통과
T0+30: ALB가 기존 Task를 deregister 시작
       → 상태가 "draining"으로 변경
       → 신규 요청 라우팅 중단
       → 기존 in-flight 요청은 최대 300초까지 유지

T+330: 300초 지나면 강제로 연결 종료, Task 종료 가능
```

블루/그린, 카나리 배포에서 가중치 라우팅 활용

- 블루/그린: 두 환경을 동시에 유지하고 LB 가중치를 0/100 → 100/0으로 순간 전환
    
    ```bash
    배포 전:  Blue(100%) ← 트래픽
              Green(0%)  ← 새 버전 배포 및 검증
    
    전환:     Blue(0%)
              Green(100%) ← 트래픽
    
    문제 발생 시: 가중치만 되돌리면 즉시 롤백
    ```
    
- 카나리: 새 버전에 점진적으로 트래픽 흘림
    
    ```bash
    1단계:  v1(95%), v2(5%)   → 메트릭 확인 (에러율, 지연시간)
    2단계:  v1(75%), v2(25%)  → 메트릭 확인
    3단계:  v1(50%), v2(50%)
    4단계:  v1(0%),  v2(100%)
    ```
    
- 두 방식 모두 가중치를 줄이는 쪽 서버는 여전히 deregistration delay 매커니즘으로 진행 중 요청을 보호
- AWS에서는 ALB의 weighted target group 기능으로 구현 가능
    
    ```bash
    Listener Rule:
      - Forward to:
          - TargetGroup-Blue:  weight 90
          - TargetGroup-Green: weight 10
    ```
    

**핵심 질문:** "Deregistration Delay를 0초로 설정하면 무엇이 깨지는가?"

1. 진행 중인 HTTP 요청이 RST로 끊김 → 클라이언트 5xx
    - LB가 즉시 "이 타겟은 더 이상 안 씀" 결정하고 기존 커넥션을 강제 종료. 응답을 절반쯤 받고 있던 클라이언트는 그대로 에러.
2. Keep-Alive 커넥션 풀에 남아있던 연결이 모두 무효화
3. 트랜잭션 원자성 깨짐
    
    ```bash
    1. 클라이언트가 송금 요청
    2. 백엔드가 DB에 출금 처리 commit
    3. 외부 시스템에 입금 알림 호출 시작
    4. 이 시점에 deregistration → 연결 강제 종료
    5. 백엔드는 죽었고, 클라이언트는 응답 못 받음
    6. 클라이언트는 "실패한 줄 알고" 재시도
    7. 출금 두 번 발생
    ```
    
4. Graceful shutdown 메커니즘 무력화
    - Springboot의 server.shutdown=gracefuldl 30초 동안 기존 요청을 마무리하려 해도, LB가 0초가 되면 연결을 끊어 버리니 30초 안에 처리하던 요청들이 응답 경로를 잃어버린다. 백엔드는 요청을 처리하고 응답을 보내려는데, 그 응답을 받아줄 LB 클라이언트 경로가 사라지는 상황이다. 백엔드 입장에서는 성공한 거래인데 클라이언트 입장에서는 실패한 거래가 되는 상황
5. 헬스체크 grace window 사라짐
