# L4 라우팅 (NLB)

stream 블록 vs http 블록 차이 이해

- stream 블록은 HTTP를 모른다

```bash
┌─ http {} ─────────────────────┐  ┌─ stream {} ───────────────┐
│ - HTTP 요청 파싱                │  │ - TCP/UDP 패킷만 봄          │
│ - URL, header, method 인식     │  │ - 페이로드 내용 모름           │
│ - location 매칭                │  │ - upstream만 매칭           │
│ - keepalive 관리               │  │ - 연결 그대로 양방향 전달       │
│ - L7 기능 풍부                  │  │ - 빠르지만 단순               │
└───────────────────────────────┘  └───────────────────────────┘
```

TCP 프록시로 MySQL/Redis 포트 분배 실습

- 기본 구조
    
    ```bash
    # /etc/nginx/nginx.conf
    
    events {
        worker_connections 1024;
    }
    
    # HTTP 블록 (L7)
    http {
        # ... 기존 L7 설정
    }
    
    # Stream 블록 (L4) — http와 동급
    stream {
        upstream mysql_backend {
            server 10.0.1.10:3306;
            server 10.0.1.11:3306 backup;
        }
        
        server {
            listen 3306;
            proxy_pass mysql_backend;
            proxy_timeout 10s;
            proxy_connect_timeout 5s;
        }
    }
    ```
    
    이렇게 하면 nginx가 3306 포트로 들어오는 TCP 연결을 백엔드 MySQL로 그대로 터널링함. nginx는 **이게 MySQL 프로토콜인지조차 모른다**. 그냥 바이트 스트림을 양방향으로 복사할 뿐.
    
- Redis 분배 실습
    
    ```bash
    stream {
        upstream redis_cluster {
            # 단순 round-robin
            server 10.0.2.10:6379;
            server 10.0.2.11:6379;
            server 10.0.2.12:6379;
        }
        
        server {
            listen 6379;
            proxy_pass redis_cluster;
            proxy_connect_timeout 1s;
            proxy_timeout 3s;
        }
    }
    ```
    
    이걸 NLB로 옮기면: NLB 생성 → Listener TCP:6379 → Target Group(Redis 인스턴스 3대) — 끝. **설정 항목이 nginx stream 블록과 거의 1:1로 매핑**
    

HTTP가 아닌 프로토콜(SMTP, gRPC, WebSocket) 처리 가능성 이해

- L7 LB는 HTTP를 알기 때문에 HTTP 아닌 트래픽은 못 보낸다. 그래서 다음 같은 경우 NLB(또는 nginx stream)가 필요
    - MySQL/PostgreSQL/Redis — 바이너리 프로토콜
    - SMTP (25, 587, 465) — 메일 서버 앞단
    - gRPC over HTTP/2 — ALB도 지원하지만 본질은 HTTP/2 + protobuf
    - WebSocket — 사실 HTTP upgrade이긴 한데 long-lived 연결이라 L4가 더 적합한 경우 있음
    - MQTT, AMQP — IoT/메시지 큐 프로토콜
    - 임의의 TCP 기반 게임 서버, IoT 디바이스 통신

소스 IP 보존(Preserve Client IP) 개념

- 기본적으로 LB가 백엔드로 연결을 만들 때 출발지 IP가 **LB의 IP로 바뀌어버려** (SNAT). 백엔드는 진짜 클라이언트 IP를 못 봄.
    
    ```bash
    stream {
        server {
            listen 3306;
            proxy_pass mysql_backend;
            
            # 클라이언트 IP 보존 옵션
            proxy_bind $remote_addr transparent;
        }
    }
    ```
    
    `transparent` 모드는 nginx가 **클라이언트인 척**하면서 백엔드에 연결함. 백엔드 입장에선 진짜 클라이언트 IP가 보인다.
    
    - nginx 프로세스가 `CAP_NET_ADMIN` 권한 필요 (보통 root)
    - 백엔드의 응답이 다시 nginx를 거치도록 라우팅/iptables 설정 필요
    
    **NLB의 경우**: Target Group을 Instance 타입으로 만들면 자동으로 client IP 보존된다. 
    
    IP 타입으로 만들면 NLB의 IP로 SNAT된다.
    
    **HTTP의 경우**는? L7이니까 `X-Forwarded-For` 헤더에 클라 IP를 박아주면 끝. 백엔드는 헤더만 보면 됨. L4는 이런 게 안 되니까 SNAT 문제가 생긴다.
    

IP Hash 기반 sticky session (L4는 쿠키를 모름)

- L4는 쿠키를 못 보니까 sticky session을 만들려면 **연결 단위 정보**로만 분배 결정을 내려야 한다.
    
    ```bash
    stream {
        upstream app_servers {
            hash $remote_addr consistent;  # 클라 IP를 해시해서 같은 서버로
            server 10.0.5.10:8080;
            server 10.0.5.11:8080;
            server 10.0.5.12:8080;
        }
        
        server {
            listen 8080;
            proxy_pass app_servers;
        }
    }
    ```
    
    consistent: 서버 한 대 빠져도 **대부분의 클라는 기존 서버로 계속 감**. 단순 모듈로 해시하면 서버 수 바뀔 때 거의 모든 클라가 재배치되는 참사가 생김 (캐시 미스 폭증)
    
    **NLB도 같은 개념**: "Source IP affinity" 옵션 켜면 동일 클라 IP는 동일 타겟으로. 단점은? 한국같이 NAT 뒤에 있는 클라가 많은 환경에선 **같은 출구 IP가 너무 많아서 부하 분산이 안 됨**. 회사 네트워크 100명이 다 한 서버로 가버림.
    

L4 LB가 ALB보다 빠른 이유 (페이로드 검사 안 함)

- HTTP 파싱 X → CPU 절약
- TLS 종단 X (passthrough면) → 암복호화 비용 0
- 헤더 검사 X, URL 매칭 X → 메모리 적게 씀
- 연결 수립 후 양방향 패킷 복사만 함 → 커널 수준 최적화 가능 (e.g. NLB는 hyperscale 처리량)

NLB는 초당 수천만 요청 처리 가능, ALB는 수십만~수백만 수준. 자리수가 다름.

**핵심 질문:** "NLB로 HTTP를 받으면 동작은 하지만, 어떤 기능들을 포기하게 되는가?"

NLB로 HTTP를 보내면 **HTTP가 그냥 TCP 위의 임의 페이로드로 취급됨**. 그래서 포기하는 게:

1. **컨텐츠 기반 라우팅 전부**
    - path 분기 X, host 분기 X, header 분기 X, method 분기 X
    - 하나의 NLB는 하나의 백엔드 풀로만 보냄. URL 보고 다른 서비스로 못 보냄.
2. **TLS 종단**
    - NLB는 TLS 종단도 가능하지만 그러면 백엔드는 다시 평문 HTTP를 받음. 통상 NLB+HTTPS면 백엔드까지 TLS passthrough(L4 그대로 통과)로 가는 경우가 많아서 백엔드 서버마다 인증서 설치해야 함.
3. **HTTP 헤더 조작**
    - `X-Forwarded-For` 자동 추가 X (그래서 IP 보존을 위해 Proxy Protocol 같은 별도 메커니즘 필요)
    - 헤더 추가/제거/수정 전부 X
4. **쿠키 기반 sticky session**
    - L4는 쿠키 모름. IP affinity로만 sticky 가능 (앞서 말한 NAT 문제 있음)
5. **HTTP 레벨 헬스체크의 풍부함**
    - NLB도 HTTP 헬스체크는 되지만, 응답 본문 검사나 복잡한 룰은 ALB가 우위
    - 기본 TCP 헬스체크는 "포트가 열려있다"만 알려줌 — 앱이 데드락 걸려도 통과
6. **WAF**
    - L4는 SQL injection 페이로드를 못 봄. 보호 못 함.
7. **리다이렉트, 고정 응답**
    - ALB는 `/old` → `/new`로 301 리다이렉트를 LB 레벨에서 처리 가능. NLB 불가.
8. **HTTP/2, gRPC 인지**
    - NLB는 그냥 TCP. ALB는 HTTP/2 처리, gRPC 메서드별 라우팅도 가능.
9. **관측성**
    - NLB 로그는 connection 단위 (출발IP, 출발포트, 도착, 바이트 수). URI, status code, User-Agent 같은 거 없음.
    - ALB 로그는 요청 단위로 풍부함.
10. **압축, 캐싱**
    - L7 LB의 영역. L4는 불가.
