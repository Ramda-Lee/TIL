# NAT Gateway로 프라이빗이 바깥 나가기

목표: 프라이빗 서브넷이 인바운드는 막으면서도 아웃바운드는 가능하게 하는 구조를 이해한다

프라이빗 서브넷의 딜레마 — 패키지 다운로드는 해야 하는데 외부 접속은 막고 싶다

- 서브넷이 퍼블릭이나 프라이빗이냐는 라우팅 테이블이 정한다. 0.0.0.0/0 경로가 IGW로 향하면 퍼블릭, 그 경로가 없으면 프라이빗이다.
- 프라이빗 서브넷의 정의: 인터넷 게이트웨로 가는 길이 없는 서브넷
- 프라이빗 서브넷의 딜레마: 인바운드는 막되 아웃바운드는 열어야 하는 상황이 생김

NAT Gateway — 아웃바운드 전용 뒷문

- NAT GW는 퍼블릭 서브넷에 존재한다. NAT GW는 자기 자신이 IGW로 나가는 경로를 가진 퍼블릭 리소스, Elastic IP를 하나 가지고 있다.
- 동작 순서
    1. 프라이빗 서브넷의 EC2가 외부로 요청 → 프라이빗 라우팅 테이블의 0.0.0.0/0이 NAT GW로 보냄
    2. NAT GW가 출발지 IP를 자기 EIP로 바꿔치기해서 GW로 내보냄
    3. 응답이 EIP로 돌아오면, NAT GW가 내부에 들고 있는 state table을 보고 원래 EC2로 돌려줌

IGW vs NAT GW 비교 — 양방향 vs 아웃바운드 전용

|  | 인터넷 게이트웨이 (IGW) | NAT 게이트웨이 |
| --- | --- | --- |
| 방향 | 양방향 (인/아웃) | 아웃바운드 전용 |
| NAT 방식 | 1:1 (퍼블릭 IP ↔ 사설 IP) | N:1 (여러 사설 IP → EIP 하나) |
| 위치 | VPC에 1개, 서브넷 무관 | 퍼블릭 서브넷 안에 배치 |
| 인바운드 | 보안그룹 열면 외부 접속 가능 | 먼저 시작한 연결의 응답만 통과 |

왜 DB는 프라이빗 서브넷에 두는가 (보안 설계)

- attack surface를 최소화하기 위해 → DB는 인터넷에 직접 접근할 일이 없다. (계층 방어)

NAT GW 가용영역(AZ) 배치 — AZ별로 둬야 하는 이유와 비용

- NAT GW는 특정 AZ의 서브넷에 생성된다.
- 흔한 실수인 AZ가 여러 개 인데 NAT GW 하나만 만들어 놓고 모든 프라이빗 서브넷이 그걸 바라보는 것
    - 문제점 1: 단일 장애점 → NAT GW가 있는 AZ-a가 죽으면 AZ-b, AZ-c의 프라이빗 인스턴스까지 전부 인터넷이 끊긴다.
    - 문제점 2: AZ간 데이터 전송 비용 → AZ-b의 인스턴스가 AZ-a의 NAT GW로 트래픽을 보내면 inter-AZ 전송 요금이 붙는다.
- 운영 권장은 AZ마다 NAT GW를 하나씩 두고, 각 AZ의 프라이빗 라우팅 테이블이 같은 AZ의 NAT GW를 가리키게 하는 것

핵심 질문: "프라이빗 서브넷의 EC2가 yum update는 되는데 외부에서 SSH 접속은 안 되는 구조는 어떻게 만들어지는가?"

- yum update는 EC2가 먼저 시작 → 라우팅 테이블 0.0.0.0/0 → NAT GW가 출발지를 EIP로 바꿔 IGW로 내보냄 → 외부 repo 응답이 EIP로 돌아옴 → NAT GW의 state table이 매칭해서 EC2로 되돌려 준다.
- SSH 안되는 이유
    - EC2에 퍼블릭 IP가 없어서 외부에서 주소를 찍을 수가 없음
    - NAT GW의 EIP로 찔러봐도, state table에 먼저 시작한 연결 기록이 없으니 매칭 실패 → 패기
    - 애초에 IGW에서 프라이빗 인스턴스로 가는 라우팅 경로 자체가 없음
