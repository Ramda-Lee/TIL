# Elastic IP, IP를 왜 고정시키는가

목표: 동적 IP의 문제를 이해하고, IP 고정이 필요한 상황을 구분한다

동적 퍼블릭 IP 문제 — EC2 재시작하면 IP가 바뀐다

- 온프레미스 환경에서는 WAS 서버에 사설 IP 넣어놓고 NAT에 공인 IP 넣어놓으면 그 IP는 물리적으로 폐기 되기 전까지 바뀌지 않는다.
- EC2의 기본 퍼블릭IP는 Stop → Start 하는 순간 변경된다. (정확히는 인스턴스를 멈추는 순간 AWS가 퍼블릭 IP를 회수해 pool에 반납한다.)
- EIP는 인스턴스가 아니라 개인의 계정에 묶이는 것이다.

EIP(Elastic IP) — 계정에 묶인 고정 퍼블릭 IP

- 온프레미스 환경과 같이 생각 한다면 EIP는 고정 공인 IP와 같다.
- EC2 인스턴스가 죽었다가 살아나도, 앞단의 공인 IP는 회선 계약처럼 유지된다.
- EIP는 IPv4만 지원, 리전에 종속되어서 다른 리전으로 옮길 수 없다.

IP 고정이 필요한 경우 (외부 시스템 방화벽 화이트리스트, NAT GW, DNS A 레코드)

- 외부 시스템 방화벽 화이트 리스트: 상대방이 특정 IP만 허용하겠다는 케이스
- NAT GW: 프라이빗 서브넷의 EC2들이 아웃바운드로 나갈 때, NAT GW를 거치면 모두 EIP 한 개로 출발지 IP가 통일된다.
- DNS A Record: 도메인의 A 레코드에 IP를 직접 박는 경우, IP가 바뀌면 레코드를 매번 갱신해야하는데 EIP는 한번 등록하면 끝이다. → ALB처럼 DNS 이름으로 붙이는 경우가 많아 줄어드는 추세

금융권 폐쇄망 연동에서 고정 IP의 의미 — 상대 기관 방화벽 등록

- 상대 기관이 IP whitelisting을 한다면 온프레미스 환경에서는 고정 IP가 있기 때문에 문제가 되지 않는다.
- 클라우드 환경에서는 인스턴스 교체마다 퍼블릭 IP가 변경되기 때문에 한계가 크다.
- 위와 같은 상황에서 NAT GW에 EIP를 붙여서 여러 대의 연동 서버가 단일 고정 IP로 나가게 설정한다.

핵심 질문: "외부 결제 기관이 우리 서버 IP를 방화벽에 등록해야 하는데, 매번 IP가 바뀌면 안 되는 상황을 어떻게 해결하는가?"

- 상대 기관이 등록하는 우리 IP는 보통 **우리가 나갈 때의 출발지 IP**고, 그걸 NAT GW의 EIP 하나로 고정하는 게 핵심이다.
- 서버를 아무리 늘리거나 교체해도 NAT GW를 거치면 출발지가 EIP로 통일되니, 상대 기관에 IP 재등록을 요청할 일이 없다.
