# 서브넷 분할과 퍼블릭/프라이빗 분리

목표: 하나의 VPC를 서브넷으로 쪼개는 기준과 AZ 배치 방식을 설계하고, 퍼블릭/프라이빗이 무엇으로 갈리는지 정의한다

서브넷 

- VPC CIDR을 더 작게 자른 단위
- 서브넷은 하나의 AZ에 종속된다.
- VPC는 리전 전체(여러 AZ)에 걸치지만, 서브넷은 못 걸친다.
- 10.30.1.0/24를 만들 때 어떤 리전에 둘지 정하고 가야한다. → 한번 정하면 그 서브넷의 인스턴스는 전부 그 AZ 안에서 산다.

AWS가 서브넷마다 예약하는 5개 IP (네트워크, VPC 라우터, DNS, 예약, 브로드캐스트) → `/24`는 **251개만** 실사용

- AWS가 5개의 IP는 예약해놓는다.
    
    ```markdown
    10.30.1.0    → 네트워크 주소        (서브넷 자체를 가리키는 주소, 호스트에 못 줌)
    10.30.1.1    → VPC 라우터           (서브넷의 기본 게이트웨이)
    10.30.1.2    → AWS DNS              (.2 리졸버, VPC DNS가 여기 산다)
    10.30.1.3    → 향후 사용 예약        (AWS가 미래 기능용으로 잡아둠)
    10.30.1.255  → 브로드캐스트 예약      (AWS는 브로드캐스트 미지원이지만 관례상 예약)
    ```
    

멀티 AZ 설계 

- 같은 역할을 궅이 2개 AZ에 두는 이유 → 가용성 때문
- AZ는 물리적으로 분리된 데이터센터여서, 정전, 네트워크, 장애와 같은 사고가 AZ 단위로 터진다. 만약 WAS 서브넷을 AZ-a에만 두면, AZ-a 데이터센터가 통째로 죽는 순간 서비스 전체가 내려감
- 같은 역할의 서브넷을 AZ-a와 AZ-c 두 곳에 깔고, 그 위에서 인스턴스를 분산시킨다.

퍼블릭/프라이빗은 서브넷의 고유 속성이 아니라 **라우팅으로 결정**된다 (STEP 3 복선)

- 퍼블릭 서브넷이라는 고유한 종류의 서브넷이 따로 있는게 아니다.
- 서브넷은 그냥 IP 대역 조각일 뿐, 퍼블릭이냐 프라이빗이냐는 그 서브넷에 연결된 라우팅 테이블이 결정

3-tier 매핑 — Web(퍼블릭) / WAS(프라이빗) / DB(프라이빗), 온프렘 DMZ·내부망 구조와 대응

```markdown
계층          서브넷 성격    이유
─────────────────────────────────────────────────────────
Web/ALB       Public        외부 사용자 트래픽을 직접 받아야 함
WAS(앱)       Private       Web만 호출, 외부 직접 노출 금지
DB(RDS)       Private       WAS만 접근, 절대 외부 노출 안 됨

# 온프엠 구조와 1:1로 대응
온프렘                          AWS VPC
─────────────────────────────────────────────
DMZ 구간 (방화벽 바깥쪽)    →   Public 서브넷  (Web/ALB)
내부망 (방화벽 안쪽)        →   Private 서브넷 (WAS)
DB존 (가장 깊은 내부)       →   Private 서브넷 (DB) + 더 엄격한 SG
```

핵심 질문: "`/24` 서브넷을 만들었는데 EC2를 256대 못 띄우는 이유는 무엇인가?"

- AWS가 5개를 가져가서 실 사용은 251개이다. → EC2 최대 251대 밖에 못 붙인다.
