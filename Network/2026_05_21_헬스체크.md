# 헬스 체크

**목표:** 백엔드 상태를 능동적으로 감지하는 헬스체크를 직접 만들고, 잘못된 헬스체크가 어떻게 전체 장애를 만드는지 이해한다

nginx의 수동적 헬스체크 (max_fails, fail_timeout) vs ALB의 능동적 헬스체크 차이

- **Nginx Open Source의 기본 방식: 수동적 헬스체크**
    - Nginx의 `max_fails`, `fail_timeout`은 보통 **실제 요청을 보내다가 실패하면 그때 판단**한다.
    
    ```bash
    upstream backend {
        server 127.0.0.1:8081 max_fails=3 fail_timeout=10s;
        server 127.0.0.1:8082 max_fails=3 fail_timeout=10s;
    }
    
    # 10초 안에 3번 실패하면
    #해당 서버를 10초 동안 죽은 서버로 간주한다.
    ```
    
- **ALB 방식: 능동적 헬스체크**
    - ALB는 실제 사용자 요청과 별개로, 정해진 주기마다 대상 서버에 직접 헬스체크 요청을 보낸다.
    - AWS ALB Target Group의 헬스체크는 `Interval`, `Timeout`, `HealthyThresholdCount`, `UnhealthyThresholdCount`, `Matcher` 같은 설정으로 동작

| **구분** | **Nginx 수동적 체크** | **ALB 능동적 체크** |
| --- | --- | --- |
| 감지 방식 | 실제 요청 실패를 보고 판단 | 주기적으로 /health 호출 |
| 사용자 요청 필요 여부 | 필요함 | 필요 없음 |
| 장애 감지 속도 | 트래픽이 있어야 감지됨 | 트래픽 없어도 감지됨 |
| 대표 설정 | max_fails, fail_timeout | interval, timeout, healthy threshold, unhealthy threshold |
| 장점 | 단순함 | 장애 감지가 명확함 |
| 단점 | 실제 요청 실패 후에야 알 수 있음 | 잘못 설정하면 헬스체크 폭탄 가능 |

헬스체크 엔드포인트 설계 (/health)

- 간단한 헬스체크 예시
    
    ```bash
    @RestController
    public class HealthController {
    
        @GetMapping("/health")
        public ResponseEntity<Map<String, String>> health() {
            return ResponseEntity.ok(Map.of("status", "UP"));
        }
    }
    ```
    

Spring Boot Actuator로 헬스 엔드포인트 노출(/actuator/health)

- Spring Boot Actuator를 사용하면 기본적으로 `/actuator/health` 엔드포인트를 제공
- Spring 공식 문서에서도 Actuator endpoint는 애플리케이션을 모니터링하고 상호작용하기 위한 기능이며, 기본적으로 health endpoint는 `/actuator/health` 경로로 매핑
- Spring Boot Actuator의 health endpoint는 애플리케이션 상태 정보를 제공하고, 세부 컴포넌트 상태도 노출할 수 있다. 예를 들어 `db`, `diskSpace` 같은 컴포넌트가 health 결과에 포함될 수 있다.
    
    ```bash
    dependencies {
        implementation 'org.springframework.boot:spring-boot-starter-actuator'
    }
    
    # 예시
    curl http://localhost:8080/actuator/health
    
    # 응답 예시
    {
      "status": "UP"
    }
    ```
    

Liveness 와 Readiness

- Liveness: 서버 자체가 살아 있는지 체크
- Readiness: 서버 프로세스는 살아있지만 트래픽을 받으면 안되는 상태
    
    ```bash
    # 실무적으로 나누는 방법
    /health/live   → 프로세스 생존 확인
    /health/ready  → 트래픽 수신 가능 여부 확인
    ```
    

헬스체크 주기/임계값/타임아웃 의미 이해 (Healthy/Unhealthy threshold, Interval)

- Interval: 몇 초마다 헬스체크를 보낼 것인지
- Unhealthy threshold: 몇 번 연속으로 실패하면 헬스체크 실패로 볼 것인지
- Healthy Threshold: 몇 번 연속으로 성공하면 다시 헬스체크 성공으로 볼 것인지

TCP 헬스체크 vs HTTP 헬스체크 차이

- TCP 헬스체크: 포트가 열려 있는지, TCP 연결이 되는 지 확인
    - 단점은 포트만 열려있어도 healthy로 판단할 수 있다. 정상 동작하는지 모른다.
- HTTP 헬스체크: HTTP 요청을 보내고 정상 응답 코드가 오는지
    - 엔드포인트 설계를 잘 못한다면 오히려 에러를 발생할 수 있다.

헬스체크에 DB 의존성을 넣으면 안 되는 이유 (전체 동시 unhealthy 함정)

- 잘못된 예시
    
    ```bash
    @GetMapping("/health")
    public ResponseEntity<String> health() {
        jdbcTemplate.queryForObject("SELECT 1 FROM DUAL", Integer.class);
        return ResponseEntity.ok("OK");
    }
    
    # 위 코드가 진행되면
    backend-1 → DB
    backend-2 → DB
    backend-3 → DB
    
    # 이로써 DB가 느려짐
    backend-1 /health 실패
    backend-2 /health 실패
    backend-3 /health 실패
    
    # ALB는 아래와 같이 판단
    backend-1 unhealthy
    backend-2 unhealthy
    backend-3 unhealthy
    ```
    
- 결론: 실제로는 백엔드 서버가 죽은 게 아니라 DB가 잠깐 느린 것뿐인데, 헬스체크가 DB에 의존했기 때문에 **전체 서버가 동시에 unhealthy**가 된다.
    - 만약 DB를 사용하고 싶다면, 로드밸런서 용이랑 운영 모니터링 용을 분리해서 사용한다.
