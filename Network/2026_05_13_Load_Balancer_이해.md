# Load Balancer 이해

단일 서버의 한계 이해 (SPOF, 처리량 한계, 무중단 배포 불가)

- SPOF (Single Point of Failure): 하나의 서버 죽으면 서비스 전체가 죽는다.
- 처리량의 한계: 인스턴스 하나가 동시에 처리할 수 있는 요청 수는 한계가 있다.
- 무중단 배포 불가: 배포 시 WAS 재시작하면 그동안 먹통이 된다.

스케일 업 vs 스케일 아웃 차이

|  | Scale Up | Scale Out |
| --- | --- | --- |
| 방식 | 서버 한 대를 더 좋게(CPU, RAM Up) | 서버 대수를 늘림 |
| 한계 | 물리적 한계 존재 | 이론상 무한 확대 가능 |
| 비용 | 고사양 장비는 가격이 비쌈 | 평범한 장비 여러 대 |
| SPOF | 존재 | 해결됨 |
| 무중단 | 어려움 | 가능 |
- 스케일 아웃을 하려면 여러 서버 앞에 트래픽을 분배해주는 로드밸런서 필요

로드밸런서의 3가지 역할 정리 (트래픽 분배 / 상태 감지 / 연결 관리)

- 트래픽분배: 들어오는 요청을 여러 서버에 골고루 나눠줌(라운드로빈, 최소 연결 수, 가중치 기반 등 알고리즘 사용)
- 상태 감지: 주기적으로 각 서버 Health Check
- 연결 관리: 클라이언트와 서버 사이에서 연결을 중재. SSL 종료, 커넥션 풀링, 세션 유지 등
- 위 3가지 역할을 다 해야 로드밸런서라는 역할로 불린다. (물리적인 장비, SW 등이 될 수 있다)

OSI 계층별 로드밸런서가 보는 정보 매핑 (L7=HTTP, L4=TCP/UDP, L3=IP)

- L7(애플리케이션 계층)
    - TCP 연결을 직접 끊어서 받은 다음, 그 안의 HTTP 요청을 파싱
    - GET /api/users HTTP/1.1, Host: google.ocm, Cookie: SESSION=….등 살펴봄
    - /api/로 시작하면 서버로, 특정 쿠키면 신규버전 서버로 등등 라우팅 가능
- L4(전송 계층)
    - TCP/UDP 헤더까지 보고, IP + Port 조합으로 분배.
    - 예를 들어 443포트로 오는 TCP 연결을 192.168.12, 11, 10 서버에 뿌리는 동작 가능
    - 안에 담긴 HTTP 내용은 못본다(payload는 바이트 덩어리, HTTPS는 암호화 되어있음)
- L3(네트워크 계층)
    - IP 헤더만 본다. → 로드밸런서보다는 라우팅에 가까움

계층이 올라갈수록 똑똑하지만 느린 트레이드오프 이해

- L4에서 L7으로 올라갈수록 로드밸런서가 할 일이 늘어난다.

|  | L4 | L7 |
| --- | --- | --- |
| 연결 처리 | 패킷 헤더만 보고 포워딩 | TCP 연결 종료 → HTTP 파싱 → 새연결 |
| CPU 부하 | 낮음 | 높음 |
| 처리량 | 매우 높음 | 상대적으로 낮음 |
| SSL | 통과 | 종료 필요 - 복호화 처리 필요 |
| 라우팅 유연성 | IP/Port 기반만 | URL, Header, Cookie 등 자유자재 |
