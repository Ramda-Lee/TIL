# L7 라우팅 (ABL)

nginx의 location은 ALB 리스너 룰과 1:1 대응

ALB가 하는 일: HTTP 요청을 파싱해서 룰에 매칭되는 타켓 그룹으로 보내는 것 (nginx의 http 블록)

```bash
ALB                          nginx
├─ Listener (80, 443)        ├─ server { listen 80; }
├─ Listener Rule             ├─ location /path { }
│   ├─ Path: /api/*          │   - path 매칭
│   ├─ Host: api.example.com │   - server_name으로 매칭
│   ├─ Header: X-Region      │   - if ($http_x_region)
│   └─ Method: POST          │   - if ($request_method)
└─ Target Group              └─ upstream { server ...; }
```

- Nginx location 매칭 우선순위

```bash
location = /exact { }      # 1. 정확히 일치
location ^~ /priority { }  # 2. prefix 우선 (정규식보다 먼저)
location ~ /regex { }      # 3. 정규식 (대소문자 구분)
location ~* /regex { }     # 4. 정규식 (대소문자 무시)
location /prefix { }       # 5. prefix 일반 (가장 흔함)

# 매칭 알고리즘

1단계: 모든 location을 훑으면서 '=' (정확 매칭) 찾기
   ├─ 찾았다 → 즉시 결정, 끝.
   └─ 못 찾음 → 2단계로

2단계: 모든 prefix location 중에서 '가장 긴 매칭' 찾기
       (= 와 ^~ 가 아닌 일반 prefix들)
   - 기억해놓고 3단계로 감 (아직 확정 X)

3단계: 2단계에서 찾은 게 '^~' modifier였나?
   ├─ 그렇다 → 정규식 검사 안 하고 그걸로 결정, 끝.
   └─ 일반 prefix였다 → 4단계로

4단계: 정규식 location들을 '설정 파일에 적힌 순서대로' 검사
   ├─ 매칭되는 정규식 발견 → 그 정규식으로 결정, 끝.
   └─ 매칭 정규식 없음 → 2단계에서 찾아둔 prefix로 결정
```

- 예시
    - 케이스
        
        ```bash
        server {
            location /api {
                return 200 "A: prefix /api\n";
            }
            
            location /api/users {
                return 200 "B: prefix /api/users\n";
            }
            
            location ~ ^/api/users {
                return 200 "C: regex ^/api/users\n";
            }
            
            location ~* \.(jpg|png)$ {
                return 200 "D: regex image\n";
            }
            
            location ^~ /static {
                return 200 "E: ^~ /static\n";
            }
            
            location = /api/users {
                return 200 "F: exact /api/users\n";
            }
        }
        ```
        
    - 결과
        
        
        | 요청 | 매칭 결과 | 이유 |
        | --- | --- | --- |
        | `GET /api/users` | **F** | 정확 매칭(=)이 최우선 |
        | `GET /api/users/123` | **C** | prefix B가 매칭되지만, 정규식 C도 매칭 → 정규식 우선 |
        | `GET /api/orders` | **A** | prefix A만 매칭 |
        | `GET /static/logo.jpg` | **E** | `^~`라서 정규식 D 검사를 안 함 |
        | `GET /img/logo.jpg` | **D** | prefix 매칭 없음 → 정규식으로 |
        | `GET /static` | **E** | `^~ /static` 매칭, 정규식 검사 skip |
- 경로 기반 분기 (/api/users → user_service, /api/orders → order_service)
    
    ```bash
    # /etc/nginx/conf.d/l7-routing.conf
    
    upstream user_service {
        server 127.0.0.1:8081;
        server 127.0.0.1:8082;
    }
    
    upstream order_service {
        server 127.0.0.1:8083;
        server 127.0.0.1:8084;
    }
    
    upstream default_service {
        server 127.0.0.1:8080;
    }
    
    server {
        listen 80;
        server_name _;
    
        # /api/users → user_service
        location /api/users {
            proxy_pass http://user_service;
            proxy_set_header X-Routed-By "path-rule:users";
        }
    
        # /api/orders → order_service
        location /api/orders {
            proxy_pass http://order_service;
            proxy_set_header X-Routed-By "path-rule:orders";
        }
    
        # 정적 파일은 별도 처리 (ALB에서 S3로 보내는 것과 동일 개념)
        location /static/ {
            root /var/www;
            expires 1d;
        }
    
        location / {
            proxy_pass http://default_service;
        }
    }
    ```
    
    - ALB매핑: Listener Rule에서 "Path is /api/users/*" → Target Group `user-tg` 와 똑같다. nginx의 `location` prefix 매칭이 ALB의 path pattern과 정확히 같은 동작
- Host 헤더 기반 가상 호스팅 (api.example.com vs admin.example.com)
    
    ```bash
    # api.example.com → API 서버군
    server {
        listen 80;
        server_name api.example.com;
        
        location / {
            proxy_pass http://user_service;
        }
    }
    
    # admin.example.com → 관리자 서버군 (완전히 다른 백엔드)
    server {
        listen 80;
        server_name admin.example.com;
        
        # IP 화이트리스트 추가 (ALB의 WAF 룰과 유사)
        allow 10.0.0.0/8; # 이 IP만 허용
        deny all;         # 나머지 모두 허용 안함
        
        location / {
            proxy_pass http://127.0.0.1:9000;
        }
    }
    ```
    
- HTTP 메서드 기반 분기 (GET은 캐시, POST는 앱 서버)
    
    ```bash
    upstream cache_layer {
        server 127.0.0.1:6081;  # Varnish 같은 캐시
    }
    
    upstream app_server {
        server 127.0.0.1:8080;
    }
    
    server {
        listen 80;
        server_name _;
    
        location /api/products {
            # GET은 캐시로, POST/PUT/DELETE는 앱으로
            if ($request_method = GET) {
                proxy_pass http://cache_layer;
                break;
            }
            proxy_pass http://app_server;
        }
    }
    ```
    
- 커스텀 헤더 기반 분기 (X-Region: kr)
    
    ```bash
    upstream kr_backend {
        server 127.0.0.1:8081;
    }
    
    upstream jp_backend {
        server 127.0.0.1:8082;
    }
    
    upstream default_backend {
        server 127.0.0.1:8080;
    }
    
    map $http_x_region $regional_backend {
        "kr"     kr_backend;
        "jp"     jp_backend;
        default  default_backend;
    }
    
    server {
        listen 80;
        
        location / {
            proxy_pass http://$regional_backend;
            proxy_set_header X-Region $http_x_region;
        }
    }
    ```
    

**핵심 질문:** "ALB가 HTTP 페이로드를 까보는 비용을 감수하면서까지 얻는 이점은 무엇인가?"

- TCP 패킷을 받아서 HTTP 프로토콜로 재조립 → CPU 소모
- TLS 종단(termination) 처리 → 암복호화 비용
- 헤더 파싱, URL 파싱 → 메모리/CPU 소모
- 결과적으로 NLB 대비 **레이턴시가 더 높고 처리량은 낮음**
