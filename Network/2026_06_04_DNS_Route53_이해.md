# DNS, Route53 이해

- DNS 조회 순서 이해 (Root → TLD → Authoritative 네임서버)
    1. 브라우저에 www.example.com을 입력
    2. Local/Recursive Resolver에서 캐시를 검색 후 캐시 있으면 반환
    3. 캐시 없으면 Root Name server에서 찾음 → .com에 대한  TLD 네임서버 반환
    4. Resolver가 TLD 네임서버(.com)에서 찾음 → example.com의  authoritative 네임서버 주소 반환
    5. Resolver가 authoritative 네임 서버에서 전체 주소 찾음 → 실제 A 레코드(IP) 반환
    6. Resolver가 결과를 캐시하고 브라우저에 반환
    
    핵심: **위임(delegation) 구조**라는 점. 각 단계는 "최종 답"이 아니라 "다음에 물어볼 곳"을 알려줄 뿐이고, authoritative에서만 진짜 답이 나옴.
    
- authoritative 네임서버가 뭔지 — Route 53이 맡는 역할
    
    Route53에서 Hosted Zone을 만들면 AWS가 4개의 NS를 할당해줌, 4개의 NS가 authoritative 네임 서버, 도메인 등록 업체에서 이 NS 4개를 등록해두면, 위 4단계 조회가 Route 53으로 도착하게 됨.
    
- 레코드 타입 매핑 (A / AAAA / CNAME / Alias / NS / MX)
    
    
    | 타입 | 역할 |
    | --- | --- |
    | **A** | 도메인 → IPv4 (`93.184.216.34`) |
    | **AAAA** | 도메인 → IPv6 |
    | **CNAME** | 도메인 → **다른 도메인 이름** (별칭). 끝에 또 조회 필요 |
    | **Alias** | Route 53 전용. 도메인 → AWS 리소스(ALB, CloudFront, S3 등). 응답 시 IP로 바로 해석 |
    | **NS** | 해당 존의 authoritative 네임서버 지정 (위임용) |
    | **MX** | 메일 서버 지정 (우선순위 값 포함) |
- CNAME vs Alias — 루트 도메인(`example.com`)에 CNAME 못 쓰는 이유
    
    CNAME: "이 이름은 사실 저 이름이야"라는 별칭. 조회하면 또 다른 도메인 이름을 돌려주므로, resolver가 한 번 더 조회해야 함 (`www.example.com → abc.elb.amazonaws.com → IP`).
    
    Alias: Route 53 내부에서 AWS가 대상 리소스의 IP를 직접 채워서 A 레코드처럼 응답함. 외부에서 보면 그냥 A 레코드.
    
    루트 도메인(`example.com`)에 CNAME을 못 쓰는 이유
    
    - DNS 표준상 CNAME은 같은 이름에 다른 레코드와 공존할 수 없음. 그런데 루트 도메인(zone apex)에는 반드시 NS 레코드와 SOA 레코드가 존재해야 함 (존의 정의 자체). 따라서 apex에 CNAME을 두면 NS/SOA와 충돌 → 표준 위반이라 못 씀.
    - `www` 같은 서브도메인은 NS/SOA가 없으니 CNAME 가능
- TTL 개념 — DNS 응답이 캐시되는 시간과 전파 지연
- 핵심 질문: "도메인을 ALB로 연결할 때 CNAME 대신 Alias를 써야 하는 이유는 무엇인가?"
    - 루트 도메인 문제 해결 — `example.com`(apex)을 ALB에 직접 연결하려면 CNAME은 위 이유로 불가능. Alias는 A 레코드처럼 동작하므로 apex에도 사용 가능. (ALB는 고정 IP가 없어 일반 A 레코드도 못 박음 → Alias가 유일한 깔끔한 답)
    - 추가 조회 없음 = 빠름 — CNAME은 resolver가 한 번 더 조회해야 하지만, Alias는 Route 53이 ALB의 현재 IP를 즉시 반환. 조회 1회 절약.
    - 비용 — Route 53에서 AWS 리소스로 향하는 Alias 쿼리는 과금되지 않음. CNAME 쿼리는 과금됨.

#### 용어 정리

Recursive Resolver (재귀 리졸버) — 실제로 발품 파는 주체. ISP가 운영하거나, 8.8.8.8(구글), 1.1.1.1(클라우드플레어) 같은 공용 리졸버. "재귀(recursive)"라는 이름은 최종 답을 얻을 때까지 대신 끝까지 물어봐 준다는 뜻. 캐시도 여기 있음.

TLD 네임서버 (Top-Level Domain**)** — `.com`, `.org`, `.kr` 같은 최상위 도메인을 관리. "`example.com`의 주인은 이 네임서버야"라고 위임 정보를 줌.

Authoritative 네임서버 (권한 있는 네임서버) — 진짜 답을 가진 서버. "authoritative = 권한 있는 = 이 도메인에 대해 내 말이 곧 진실"이라는 뜻. Route 53이 맡는 자리가 바로 여기. 위의 Root/TLD는 길만 알려주지만, 여기서만 "IP는 93.184.216.34야"라는 최종 답이 나옴.
