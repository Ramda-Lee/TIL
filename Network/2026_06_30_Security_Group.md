# Security Group
목표: SG가 ENI 단위로 동작하는 Stateful 방화벽임을 이해하고, 응답 패킷이 자동 허용되는 원리를 설명할 수 있다

Security Group - 인스턴스(서버) 하나하나를 감싸는 방화벽.

SG는 서브넷이 아니라 **ENI(인스턴스)에 붙는다**

- 라우트 테이블과 NACL은 서브넷 단위로 동작하는데, SG는 Elastic Network Interface 단위로 동작한다.
    
    ```markdown
    서브넷 (10.30.3.0/24, DB 계층)
    ├─ DB 인스턴스 #1  ← ENI ← SG-DB 적용
    ├─ DB 인스턴스 #2  ← ENI ← SG-DB 적용
    └─ (모니터링) 인스턴스 #3 ← ENI ← SG-Monitoring 적용  ← 같은 서브넷이어도 다른 SG 가능
    ```
    

**Stateful** — 인바운드를 허용하면 그에 대한 아웃바운드 응답은 자동 허용 (역방향도 성립)

- Stateful은 연결을 시작한 방향과 무관하게, 그 연결의 응답은 자동 허용
- 정방향: 인바운드 → 아웃바운드 응답 자동 허용
    
    ```markdown
    WAS가 DB로 연결 (DB 입장에선 인바운드)
      → DB SG 인바운드 1521 허용 → 통과
      → DB가 응답 (DB 입장에선 아웃바운드)
      → 아웃바운드 규칙 안 봐도 자동 허용 ★
    ```
    
- 역방향: 아웃바운드 시작 → 인바운드 응답 자동 허용
    
    ```markdown
    WAS가 외부 API로 요청 (WAS 입장에선 아웃바운드)
      → WAS SG 아웃바운드 허용(기본 전체 허용) → 나감
      → 외부 API가 응답 (WAS 입장에선 인바운드)
      → 인바운드 규칙에 아무것도 없어도 자동 허용 ★
    ```
    

**Allow 규칙만 존재** — Deny를 못 씀, 순수 화이트리스트 방식

- SG에는 허용(Allow) 규칙만 존재하고, 차단(Deny) 규칙을 못 쓴다.
- 규칙에 명시되지 않은 건 전부 자동 차단되는 화이트 리스트 방식
- 특정 대역 차단이 필요하다면 NACL(Deny 규칙)을 써야 한다.

소스로 IP뿐 아니라 **다른 SG를 지정 가능** — "WAS SG에서 오는 것만 DB SG에 허용" 패턴

- SG 규칙의 소스에 IP/CIDR뿐 아니라 다른 SG를 지정할 수 있다.
    
    ```markdown
    SG-DB 인바운드 규칙:
      포트 1521, 소스 = SG-WAS    ← IP가 아니라 SG를 가리킴
    
    의미: "SG-WAS가 붙은 인스턴스에서 오는 1521 트래픽만 허용"
    ```
    
- WAS 인스턴스의 IP가 바뀌거나 오토스케일링으로 늘어나도 규칙을 안 고쳐도된다.

기본 SG 동작 — 아웃바운드 전체 허용, 인바운드 전체 차단

- 새로 만든 SG 초기 상태는 인바운드는 전부 차단, 아웃바운드는 전부 허용이다.

핵심 질문: "DB SG에 인바운드 1521만 열고 아웃바운드는 아무것도 안 열었는데, WAS가 쿼리 응답을 정상적으로 받는 이유는?"

- SG이 stateful이기 때문이다.
