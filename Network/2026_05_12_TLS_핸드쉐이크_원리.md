# TLS 핸드쉐이크 원리

HTTP vs HTTPS 차이 (평문 vs 암호화)

- HTTP는 TCP 위에서 그냥 텍스트를 주고 받는다. 와이어샤크 켜면 `POST /login` 요청의 `password=1234`가 그대로 보이는데 중간에 있는 누구든 (카페 와이파이 운영자, ISP, 회사 프록시  등) 다 읽을 수 있고, 심지어 내용을 **변조**할 수도 있다. (MITM, Man-in-the-Middle).
- HTTPS는 HTTP 메시지를 **TLS** 라는 보안 계층으로 감싼 것.
- TLS는 3가지를 보장
    - **기밀성 (Confidentiality)**: 암호화되어 있어서 중간자가 못 읽음
    - **무결성 (Integrity)**: MAC(Message Authentication Code)으로 변조 감지
    - **인증 (Authentication)**: 인증서로 "이 서버가 진짜 xxxx.com 맞다"를 검증

대칭키 / 비대칭키 암호화 차이와 각각의 쓰임새

**대칭키 (Symmetric)**: 암/복호화에 **같은 키** 사용. AES가 대표적.

- 장점: 빠름. AES-256 같은 건 CPU에 AES-NI 명령어까지 있어서 거의 공짜에 가까운 속도.
- 단점: 키를 어떻게 안전하게 상대방에게 전달하지? (= 키 교환 문제)

**비대칭키 (Asymmetric)**: 공개키와 개인키 **쌍**으로 동작. RSA, ECC가 대표.

- 공개키로 암호화 → 개인키로만 복호화
- 개인키로 서명 → 공개키로 검증
- 장점: 키를 미리 공유할 필요 없음. 공개키는 그냥 뿌려도 됨.
- 단점: **느림**. RSA 2048비트는 AES 대비 수백~수천 배 느립니다.

TLS 핸드셰이크 흐름 이해 (ClientHello → 인증서 → 키 교환 → 대칭키 통신)

**1) ClientHello**: 클라이언트가 "나는 이런 TLS 버전들 지원하고, 이런 암호 스위트(cipher suite) 쓸 수 있어"라고 알림. 동시에 랜덤 난수(Client Random)도 보냄.

**2) ServerHello + Certificate**: 서버가 "그럼 이 암호 스위트로 가자"고 선택한 뒤, 자기 인증서를 같이 보냄. 이 인증서 안에 서버의 **공개키**가 들어 있음. 서버 랜덤도 함께.

**3) 인증서 검증**: 클라이언트는 받은 인증서를 검증.

- 만료 안 됐는지
- 도메인이 일치하는지
- **CA가 서명한 게 맞는지** (브라우저/OS에 내장된 Root CA의 공개키로 서명 검증)
- 폐기되지 않았는지 (OCSP/CRL)

**4) 키 교환**: 클라이언트가 **pre-master secret**이라는 랜덤값을 만들어서, **서버의 공개키로 암호화**해서 보냄. 서버의 개인키를 가진 사람만 이걸 풀 수 있다

**5) 세션 키 생성**: 양쪽 다 (Client Random + Server Random + pre-master secret)을 조합해서 같은 대칭키 (session key)를 만든다. 이걸 네트워크로 주고받은 적이 없다는 게 핵심.

**6) 대칭키로 실제 통신**: 이후 모든 HTTP 메시지는 이 AES 세션 키로 암호화.

CA(인증기관)의 역할과 인증서 구조 이해

CA는 "이 공개키는 진짜 naver.com 거 맞아요"를 보증하는 **제3자 신뢰 기관**

**왜 필요한가?** 비대칭키만 있으면 MITM이 막히지 않는다. (공격자가 가짜 공개키를 끼워넣을 수 있음) CA가 서명한 인증서는 위조가 불가능해서 이걸 막는다.

**인증서 체인 (Chain of Trust)**:

```jsx
Root CA (브라우저/OS에 내장, 셀프 서명)
   ↓ 서명
Intermediate CA
   ↓ 서명
Server Certificate (naver.com)
```

브라우저는 Root CA의 공개키를 미리 가지고 있어서 서버가 보낸 인증서의 CA 서명을 검증할 수 있다.
