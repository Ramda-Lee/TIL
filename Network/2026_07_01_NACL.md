# NACL

목표: NACL이 서브넷 단위 Stateless 방화벽임을 이해하고, 왜 ephemeral 포트를 따로 열어야 하는지 설명할 수 있다

NACL은 **서브넷 경계**에서 동작 → 그 서브넷의 모든 인스턴스에 일괄 적용

- NACL은 서브넷 경계에서 동작하는 방화벽이다. → 서브넷을 드나드는 모든 트래픽이 여기를 통과

**Stateless** — 인바운드/아웃바운드를 각각 명시해야 함, 응답 패킷이 자동으로 허용되지 않음

- SG과 결정적 차이 - 상태를 기억하지 않는다.
- 들어온 연결을 기록해두지 않아서, 나가는 패킷도 아웃바운드 규칙으로 처음부터 다시 검사

```markdown
SG (Stateful):
  인바운드 허용 → 그 응답은 아웃바운드 규칙 안 봐도 자동 통과 (편함)

NACL (Stateless):
  인바운드 허용해도 → 응답은 아웃바운드 규칙을 또 통과해야 함 (직접 열어야 함)
  → 그래서 ephemeral 포트를 아웃바운드에 열어주는 작업이 필요
```

번호(rule number) 순서대로 평가, **첫 매치에서 결정** (낮은 번호 우선)

- NACL 규칙은 번호가 매겨져 있고, 낮은 번호부터 순서대로 평가하다가 처음 매치되는 규칙에서 즉시 결정된다. → 매칭되면 그 아래 규칙은 안본다.
- SG와 차이점 - SG는 모든 규칙을 종합해서 하나라도 허용이면 허용, NACL은 번호순 + 첫 매치 승리 순서이다.

**Allow와 Deny 둘 다 가능** — SG와 결정적으로 다른 점

- SG는 Allow만 가능하지만 NACL은 Deny도 쓸 수 있다.
- 예를 들어 악성 IP 한 대만 막고 나머지는 허용하려면 NACL로만 할 수 있다.

**ephemeral 포트(32768–65535)** — 응답 트래픽이 돌아오는 포트, 인바운드/아웃바운드에 열어줘야 통신 완성

- 클라이언트가 서버에 접속할 때, 자기 쪽 출발지 포트는 OS가 임의의 높은 번호로 자동 할당. 이 임시 포트 범위가 ephemeral 포트다

```markdown
클라이언트가 웹서버:80에 접속할 때 패킷:
  출발지: 클라이언트IP : 51000   ← OS가 자동으로 고른 임시 포트(ephemeral)
  목적지: 서버IP      : 80       ← 우리가 아는 그 포트

서버가 응답할 때는 출발지/목적지가 뒤바뀜:
  출발지: 서버IP      : 80
  목적지: 클라이언트IP : 51000   ← 아까 그 임시 포트로 돌려줌
```

- 서버 → 클라이언트 응답은 80이 아니라 51000을 목적지로 나간다.
- NACA 아웃바운드가 80만 열려 있고 ephemeral 범위가 닫혀 있으면 이 응답이 막힌다.

기본 NACL은 전체 허용 / 새로 만든 커스텀 NACL은 기본 전체 거부

핵심 질문: "NACL 인바운드에 80은 열었는데 클라이언트가 웹 응답을 못 받는다. 무엇을 빠뜨렸나?"
