# DNS 쿼리 원리

DNS가 필요한 이유 이해

- 사람이 기억할 수 있는 이름으로 접근
- 백엔드 IP가 바뀌어도 클라이언트는 영향 없음
- 한 도메인 → 여러 IP (로드밸런싱, GeoDNS)
- 한 IP → 여러 도메인 (가상 호스팅)

DNS 조회 6단계 흐름 이해 (브라우저 캐시 → OS → 로컬 DNS → Root → TLD → Authoritative)

1. 브라우저 캐시 
2. OS 캐시 (hosts)
3. 로컬 DNS (아래 과정은 전부 로컬 DNS가 진행)
4. Root Name Server
5. TLD (Top-Level Domain)
6. Authoritative Name Server

주요 레코드 타입 정리 (A / CNAME / MX / NS / TTL)

| 타입 | 역할 | 예시 |
| --- | --- | --- |
| **A** | 도메인 → IPv4 | `example.com → 93.184.216.34` |
| **AAAA** | 도메인 → IPv6 | `example.com → 2606:2800:220:1::` |
| **CNAME** | 도메인 → 다른 도메인 (별칭) | `www.example.com → example.com` |
| **MX** | 메일 서버 지정 (우선순위 포함) | `example.com → 10 mail.example.com` |
| **NS** | 해당 도메인의 권한 서버 지정 | `example.com → ns1.example.com` |
| **TXT** | 임의 문자열 (SPF, DKIM, 도메인 소유 검증) | `v=spf1 include:_spf.google.com ~all` |
| **TTL** | 레코드 캐시 유효 시간(초) | `300` = 5분 |
- **루트 도메인(`example.com` 자체)에는 CNAME을 못 쓴다**. RFC 1912에 따라 NS/SOA와 충돌. 그래서 AWS는 Route53의 ALIAS 레코드 같은 우회 수단을 제공
